# Возврат knowledge-эндпоинта (wiki/ai-chat интеграция)

## Контекст

Изначально `/chat` шёл через router (LLM-классификатор `data | knowledge`),
и при `knowledge` ходил в внешний сервис `ai-chat` (Uniss1/ai-chat) для ответа
по базе знаний (Wiki.js + векторный поиск).

Сейчас отключён, потому что:

- **Латентность.** Router добавлял ~1–3 сек к каждому запросу, причём в 95 %
  случаев ответ — `data`. Накладные расходы того не стоят.
- **Шум при отладке.** Tool calling ещё нестабилен на Gemma 4 E2B; добавление
  второго LLM-шага усложняло поиск причин ошибок.
- **Фокус.** Сейчас приоритет — добиться надёжного tool calling и сборки
  запросов на стороне 1С. Knowledge — отдельная история.

## Что осталось после отключения

- `POST /knowledge` — заглушка, возвращает 503 «temporarily disabled».
- В коде `wiki_client.py` и `router.py` удалены — при возврате восстановить
  из git history (commit `cb34c00` ещё их содержит).
- Конфиг `WIKI_BASE_URL` и `wiki_timeout` оставлены в `.env`/`config.py` —
  пусть лежат, не мешают.

## План возврата

1. **Решить use-case.** Когда понадобится knowledge — оформить как отдельный
   эндпоинт `POST /knowledge` (без LLM-роутера), который фронт вызывает
   явно (отдельная вкладка/кнопка в виджете). Никакого классификатора в
   `/chat`. Пользователь сам выбирает: «спросить про данные» vs «спросить
   про вики».
2. **Восстановить `wiki_client.py`** из git: `git show <commit>:api/wiki_client.py`.
3. **Реализовать `POST /knowledge`** — принимает `{message, history?}`,
   зовёт `ask_knowledge_base`, возвращает `{answer, sources, from_cache}`.
4. **UI:** добавить переключатель «Данные / Знания» во фронт (`web/index.html`,
   виджет, расширение).
5. **Кеш:** опционально — отдельная таблица `knowledge_cache` или флаг
   `intent` в существующем кеше.

## Что НЕ возвращать

- LLM-роутер `classify_intent` — намеренно убран, не нужен. Выбор
  «data vs knowledge» делает пользователь, а не модель.
